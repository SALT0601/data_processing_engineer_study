요구사항 개발 프로세스 - 출석명확
- 도출(Elicitation) -> 분석(Analysis) -> 명세(Specification) -> 확인(Validation)
- 개발 프로세스 전 타당성 조사(Feasibility Study) 가 선행되어야 함
- 요구공학의 요소 (Requirements Engineering 
                 - 무엇을 개발해야하는지 요구사항 정의, 분석 및 관리 프로세스를 연구하는 학문
                 - 요구사항 변경의 원인과 처리 방법을 이해하고 관리 프로세스의 품질 개선으로 프로젝트 실패를 최소하는 것이 목표)


요구사항 도출(Requirement Elicitation, 요구사항 수집)
- 서로 의견 교환하여 식별하고 이해하는 과정
- 개발자와 고객 사이, 이해관계자(Stakeholder)가 식별
- 소프트웨어 개발 생명 주기(SDLC)동안 지속적으로 반복
- 주요 기법
=> 청취와 인터뷰, 설문, 브레인스토밍(3인 이상 의견교환), 워크샵, 프로토타이핑(프로토타입으로 명세서 산출, 그림 그리기), 유스케이스(사용자의 기능 요구사항)

요구사항 분석(Requirement Analysis) 
- 요구사항 중 이해되지 않는 부분을 걸러내기 위한 과정 => 타당성 확보
- 비용과 일정 등 제약 설정
- 상충되는 것 중재
- 대표 도구 => 자료흐름도(DFD), 자료사전(DD)

요구사항 명세(Requriement Specification)
- 분석된 요구사항을 바탕으로 모델 작성, 문서화
- 기능 : 빠짐없이 기술, 비기능 : 필요한 것만 기술
- 구체적인 명서 - 소단위 명세서(Mini-Spec) = 모듈명세서

요구사항 확인(Requirement Validation, 요구사항 검증)
- 요구사항 명세서가 정확하고 안전하게 작성되었는지 검토
- 이해관계자들이 검토
- 형상관리(SCM) 수행 - 개발 과정에서 만들어지는 형상들에 대한 변경 사항을 관리하는 활동

요구사항 명세 기법
- 정형 명세 기법 
=> 작성법 : 수학을 기반으로 정형형화된 표기법이 있어 특정 규칙이 있음
     특징 :정확하고 간결, 작성자에 관계없이 일관성이 있어 완전성 검증이 가능하나 표기법이 어려워 사용자가 이해하기 어려움.
     종류 : VDM, Z, Petri-net, CSP등

- 비정형 명세 기법
=> 작성법 : 상태,기능,객체 중심으로 일반 명,동사 등 자연어를 기반으로 서술
     특징 : 자연어 사용으로 작성자에 따라 일관성 떨어지고 해석 달라지지만 내용이 이해하기 쉬워 의사소통 용이
     종류 : FSM, Decision Table, ER모델링, State Chart(SADT) 등
