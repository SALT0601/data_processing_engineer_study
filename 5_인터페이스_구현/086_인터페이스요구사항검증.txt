인터페이스 요구사항 검증
- 사용자들의 요구사항이 요구사항 명세서에 정확하고 완전하게 기술되었는지 검토하고 개발 범위의 기준인 베이스라인 설정
- 즉 요구사항 명세서에 명시되어 있는 사용자의 요구사항들이 실제로 실현 가능한지 확인하는 단계
- 수행 순서
-> 요구사항 검토 계획 수립(검토 기준 및 방법, 참여자, 체크리스트, 관련자료, 일정 등 검토 계획 수립
-> 검토 및 오류 수정(인터페이스 요구사항 명세서 검토 및 오류 목록과 시정 조치서 작성)
-> 베이스라인 설정(검증된 인터페이스 요구사항을 승인 받은 후 요구사항 명세서의 베이스라인 설정)

요구사항 검증 방법
- 요구사항 검토 : 요구사항 명세서의 오류 확인 및 표준 준수 여부 등 결함 여부를 검토 담당자들이 수작업으로 분석
-> 동료 검토 : 작성자가 명세서 내용을 직접 설명
-> 워크 스루 : 명세서를 미리 배포
-> 인스펙션 : 검토 전문가들이 명세서 확인
- 프로토타이핑 : 견본품을 통한 결과물 예측
- 테스트 설계 : 테스트 케이스 생성해 검토
- CASE 도구 : 일관성 분석통해 추적, 분석, 관리, 표준 준수 여부 확인

인터페이스 요구사항 검증의 주요 항목
- 완전성 : 사용자의 모든 요구사항 누락 안되고 완전하게 반영
- 일관성 : 요구사항 모순되거나 충돌되는 점 없이 일관성 유지
- 명확성 : 모든 참여자가 요구사항 명확히 이해 가능
- 기능성 : 요구사항이 어떻게 보다 무엇을에 집중
- 검증 가능성 : 요구사항이 사용자의 요구를 모두 만족하고 개발된 소프트웨어가 사용자의 요구 내용과 일치
- 추적 가능성 : 요구사항 명세서와 설계서 추적 가능
- 변경 용이성 : 요구사항 명세서의 변경이 쉽도록 작성
